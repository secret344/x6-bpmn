
# Pool / Lane 新方案可行性分析

## 文档定位

这份文档只讨论新方案本身是否可行，以及它成立所需的前提条件。

这里不再以当前实现为出发点，也不讨论如何在现有代码上最小修补。重点是判断这套模型本身是否足够稳定，是否值得作为后续重构方向。

这里的“内部节点不能暴露到 Pool 外侧”仍然按编辑器交互策略讨论，不直接上升为 BPMN 规范结论。

## 结论

新方案是可行的，而且比把 Pool、Lane、删除修复拆成多段局部规则更适合作为长期模型。

可行的核心原因有三点：

- 约束域可以被清晰拆开，不需要再让兄弟 Lane 的内容互相串联。
- preview、commit、delete 可以统一落到同一套几何收敛模型里。
- 这套模型天然可以向 `childLaneSet` 扩展，不需要未来再推翻一次基础语义。

这套模型还可以进一步收紧成一个更明确的执行原则：尺寸变化默认只在当前层级收敛。只有两类操作例外：

- 直接删除整个节点。
- 直接拖拽 Pool 外边界。

即使是 Pool 外边界拖拽，也不需要在拖拽过程中持续重排，只需要在拖拽结束后找到贴着该 Pool 边界的 Lane，再做一次收敛调整。

真正的难点不在“能不能做”，而在“边界归属是否先定义清楚”。如果边界归属先定清楚，这个方案可以稳定实现；如果边界归属不清楚，后续会反复出现局部正确、整体打架的问题。

结合 `bpmn-js` 的现有实现来看，这份方案里有两类内容：

- 一类与 `bpmn-js` 基本一致，可以直接作为参考依据。
- 一类是相对 `bpmn-js` 的有意简化，需要在文档里明确写成“设计选择”，而不是“参考实现事实”。

## 新方案的核心模型

### 1. 统一容器模型

- Pool 是泳道容器。
- Lane 也是泳道容器。
- `childLaneSet` 不是特殊补丁，而是 Lane 下面继续嵌套一层同类容器集合。

这意味着后续不应该把 Pool 和 Lane 当成两套不同体系，而应该统一成“容器节点 + 同层 lane 集合 + 内部内容”的模型。

### 2. 统一约束模型

新方案成立的前提，是把约束分成三类，并且始终分开计算：

- Pool 外边界约束：决定容器外轮廓还能不能继续缩小。
- 同层 Lane 分隔线约束：决定共享边界还能不能继续移动。
- 当前容器自身内容约束：决定当前容器内部可压缩到什么程度。

只要这三类约束不混写，新方案就是可推导的。反过来，一旦把它们混成一个“最小尺寸”概念，模型就会重新失稳。

### 3. 统一阶段模型

所有交互都应被拆成三个阶段：

- preview：只算合法 ghost。
- commit：写入当前层级的最终几何。
- reconcile：默认只在当前层级内分配剩余空间。

删除 Lane 也不应被看成例外流程，而应视为一种特殊的 reconcile：先释放空间，再按同层规则重新分配。

## 为什么这个方案可行

### 1. 约束推导是局部闭合的

对于任意一条 Lane，它真正需要知道的信息只有三块：

- 当前容器自己的内容边界。
- 当前容器所在同层集合的相邻 Lane 最小尺寸；如果相邻 Lane 是嵌套 Lane，则取其内部与当前边界相邻的贴边子 Lane。
- 当前层级的外边界归属。

这意味着大部分交互都可以在当前层级闭合推导，不需要一开始就扫描整棵树。默认只在当前层级内收敛；只有用户直接操作 Pool 外边界时，才进入外层边界变化语义。

进一步说，Pool 外边界拖拽也不需要在拖拽过程中持续把影响扩散到整组 Lane。只要在拖拽结束后，定位贴着该 Pool 边界的 Lane，再做一次统一调整即可。

这正是新方案可行的关键，因为它避免了“所有交互都变成全局重算”。

需要注意的是，这里与 `bpmn-js` 存在一个明确差异：`bpmn-js` 在 balanced resize 时，会从当前 lanes root 递归收集全部 Lane，再用几何邻接阈值判断哪些 Lane 需要一起调整，而不是只按当前层级局部闭合处理。因此，这份文档里的“默认只在当前层级内收敛”应视为有意偏离 `bpmn-js` 的简化方案。

### 2. 删除、拖拽、缩放可以共用一套规则

新方案不是为 resize 单独设计，再为 delete 额外补规则，而是把三者放进同一条语义线上：

- 拖拽共享边界，本质上是重新分配同层空间。
- 缩放外边界，本质上是重新决定容器轮廓。
- 删除 Lane，本质上是释放一段同层空间，再重新分配。

只要这三种操作最终都落回“当前层级空间如何分配”这个问题，模型就不会越长越碎。

### 3. 对 `childLaneSet` 友好

如果现在就把 Lane 当成可继续拥有 lane 集合的容器，新方案未来扩到 `childLaneSet` 时，只是多一层递归，而不是再发明一套新语义。

这点很重要。因为如果现在的模型只适用于平铺 Lane，将来一旦进入嵌套，就必须重新定义谁拥有外边界、谁负责分配空间、谁向谁传播约束。那时重构成本会更高。

## 已明确的前提

### 1. 边界归属

边界归属已经可以先固定为：

- Pool 外边界属于当前 Pool。
- 同层 Lane 的共享分隔线属于同层 lane 集合。
- Lane 的内部空间下限属于当前 Lane 自己。
- 如果相邻 Lane 是嵌套 Lane，则共享分隔线对应的合法最小高度由该相邻 Lane 内部当前方向上真正贴边的子 Lane 决定。

补充边界如下：

- 当前模型只考虑单个 Pool，不存在多个 Pool 之间的联动，也不存在 Pool 嵌套。
- Lane 不允许移动，只允许 resize，因此不需要为 Lane 单独设计平移语义。
- Lane 拖拽只需要考虑自身最小高度限制，以及拖拽方向上相邻 Lane 的最小高度限制。
- 如果拖拽方向上的相邻 Lane 是嵌套 Lane，则这里的“相邻 Lane 最小高度限制”应继续下钻到该嵌套 Lane 内部当前方向上真正贴边的子 Lane。
- 其他 Lane 默认不参与约束计算；只有当被拖拽的边本身就是 Pool 外边界时，才进入 Pool 外边界约束。

这里的“真正贴边”不需要设计得太复杂，直接按方向取最外侧子 Lane 即可。例如向上拖拽时，只需要找到上方相邻 Lane 内部最底部的那个子 Lane；向下拖拽时，只需要找到下方相邻 Lane 内部最顶部的那个子 Lane。

左右方向与上下方向完全镜像处理：向左拖拽时取左侧相邻 Lane 内部最右侧的子 Lane；向右拖拽时取右侧相邻 Lane 内部最左侧的子 Lane。

如果这个归属不写死，后面所有 preview 和 commit 都会出现语义摇摆。

### 2. 删除后的空间分配策略

删除 Lane 后，释放出的空间可以直接定为：

- 两侧都存在时，固定按 1:1 分配给两侧相邻 Lane。
- 只有一侧存在时，全部交给该侧。

删除当前 Lane 时，不需要再额外设计复杂的几何判定，直接调整相邻 Lane 的高度即可。如果接收空间的相邻 Lane 本身是嵌套 Lane，则按删除这一侧的方向，从父到子逐层找到与当前侧相邻的 Lane，并逐层调整高度。

这不是实现细节，而是模型语义。这个规则一旦固定，删除行为就可以从“补丁修复”转成“规则驱动”。

这一点与 `bpmn-js` 的删除行为是基本一致的：`bpmn-js` 在删除 Lane 时，双侧都有相邻 Lane 则按一半补偿，只有单侧存在时则全部补给该侧。

### 3. 删除与平移语义

这里可以直接固定为：

- Lane 不允许移动，所以不存在 Lane 拖拽导致的平移问题。
- 删除 Lane 时，只做同层空间重新分配，不引入“整体顶上去”的额外语义。
- 删除 `childLaneSet` 时，子树直接一起删除，不引入额外搬迁逻辑。

这个约束的价值，在于避免删除动作继续叠加新的层层平移规则，也避免把“移动 Lane”这种并不存在的交互引入模型。

### 4. 跨层传播条件

跨层传播也可以先明确收紧为：

- 默认不向上传播。
- 每一层都优先在本层完成边界约束和空间分配。
- 只有用户直接拖拽 Pool 外边界，或者直接删除整个节点时，才允许影响传导到贴着 Pool 边界的 Lane。
- 即使发生这种传导，也只需要在交互结束后统一调整一次，不需要在拖拽过程中持续重排。

如果被影响的贴边 Lane 本身还是嵌套 Lane，则传导也只继续到该侧方向上的贴边子 Lane，不扩散到整组子 Lane；删除场景下则按删除侧从父到子逐层处理相邻 Lane。

删除 `childLaneSet` 在模型上等价于删除一个子层 Lane，后续可以再对照 bpmn-js 做一次行为校验。

从 `bpmn-js` 的结构实现看，`lane.childLaneSet` 确实按 Lane 树继续递归处理，导入和遍历上都把它当成 Lane 层级的一部分；但在 resize 约束上，`bpmn-js` 仍然更偏向“从 lanes root 收集后按几何关系判断”，而不是完全只走你这里定义的“当前层级 + 贴边链”。

## 最大风险点

### 1. 共享边界和外边界混写

这是最容易把新方案做坏的地方。

一旦把“Pool 外边界”和“Lane 分隔线”当成同一种边界处理，兄弟 Lane 串联、删除后约束错位、嵌套时父子相互打架这些问题都会回来。

### 2. preview 和 commit 语义不一致

如果 preview 按局部规则算，commit 却按另一套全局规则落地，用户会看到 ghost 合法但提交后跳变。

所以新方案必须要求 preview 和 commit 使用同一套约束语义，只允许阶段不同，不允许规则不同。

上面两项是仍然需要保留的实质风险。其他像“删除语义不稳定”或“跨层传播过度”，在当前文档已经把规则收紧之后，已经更适合写成实现约束，而不是继续单列成风险点。

## 与 `bpmn-js` 对照后的调整建议

### 可以直接保留的部分

- 删除 Lane 双侧时固定 1:1、单侧缺失时全给另一侧，这一点与 `bpmn-js` 的补偿逻辑一致。
- `childLaneSet` 作为 Lane 树递归的一部分，这一点与 `bpmn-js` 的导入和遍历方式一致。
- 把嵌套场景继续落回 Lane 语义，而不是发明独立容器语义，这个方向与 `bpmn-js` 一致。

### 需要明确写成“有意偏离”的部分

- 当前文档主张：尺寸变化默认只在当前层级收敛。
	`bpmn-js` 现状：balanced resize 会从 lanes root 递归收集全部 Lane，再按几何邻接阈值决定哪些 Lane 跟着调整。
- 当前文档主张：嵌套时优先按删除侧或拖拽侧，从父到子逐层只处理贴边 Lane。
	`bpmn-js` 现状：更偏向先把所有相关 Lane 收集出来，再按几何贴边关系做补偿与约束。

这两条不是文档错误，但建议在实现阶段明确标注为“相对 `bpmn-js` 的简化策略”，否则后续读代码的人容易误以为这就是 `bpmn-js` 的既有规则。

当前决策可以进一步明确为：首版实现先采用本文档里的局部收敛方案，不主动兼容 `bpmn-js` 的全量递归 resize 模式；只有在真实实现或测试中出现当前方案无法稳定覆盖的边界场景时，再回头引入 `bpmn-js` 的对应处理思路。

## 建议的方案边界

如果要推进这套新方案，我建议先把边界收紧到下面这组最小可行规则：

- preview 只负责算合法 ghost，不做真实节点修复。
- commit 只先落当前层级，不直接跨层全量重排。
- reconcile 优先在当前层级内解决，默认不向父层传播。
- Pool 外边界拖拽的影响只在拖拽结束后收敛到贴边 Lane，不做持续重排。
- 删除 Lane 先走同层空间再分配，再走统一约束修正；两侧都存在时固定按 1:1 分配，如果接收方是嵌套 Lane，则按删除侧从父到子逐层调整相邻 Lane。
- `childLaneSet` 延续同一套规则，不单独发明新流程。

这套边界足够保守，但已经能支撑平铺 Lane 和未来嵌套 Lane 的统一模型。

## 推荐推进顺序

### 第一阶段：平铺 Lane 模型定型

- 先把 Pool 外边界、Lane 分隔线、Lane 自身内容下限这三类约束彻底定清楚。
- 只讨论单层 Lane，不进入嵌套。
- 先把 preview、commit、reconcile 三阶段术语固定下来。

### 第二阶段：删除语义定型

- 确认删除后的空间分配规则。
- 确认删除后不引入额外平移语义。
- 确认只有删除整个节点时，影响才允许传导到贴着 Pool 边界的 Lane。

### 第三阶段：`childLaneSet` 递归化

- 把 Lane 提升为可继续持有 lane 集合的容器。
- 验证跨层传播的截断条件。
- 验证删除子 Lane 时继续保持“直接删除、不搬迁”的规则是否足够稳定。

这里的尺寸约束可以直接先按一条明确原则实现：只要 Lane 的尺寸限制已经成立，`childLaneSet` 的最小尺寸就必然由其内部每一个最后一层 Lane 共同决定。

这个顺序的好处是，先把平铺语义定死，再把嵌套作为同模型递归，而不是同时讨论所有层级，导致问题互相污染。

## `childLaneSet` 实现计划

### 设计目标

- 支持 Lane 下继续挂载子 LaneSet。
- 让 Pool、Lane、`childLaneSet` 共用一套约束与重排模型。
- 保持同层约束优先，避免跨层 sibling 串联。

### 实现思路

1. 先查询 BPMN 规范中 `laneSet` 和 `childLaneSet` 的结构约束。
2. 对照 bpmn-js 的实现确认层级、命中测试、重排和导入导出行为。
3. 在内部模型上，把 Pool 和 Lane 都视为可拥有 lane 集合的容器节点。
4. preview 阶段只计算当前层级的合法 ghost，不跨层联动其他 lane 集合。
5. commit 阶段从当前层级开始收敛；只有直接操作 Pool 外边界或直接删除整个节点时，才在结束后把影响投影到贴边 Lane。
6. 删除子 Lane 时，也优先在当前 `childLaneSet` 内完成空间分配；两侧都存在时固定按 1:1 分配，如果相邻接收方还是嵌套 Lane，则按删除侧从父到子逐层调整相邻 Lane，不额外引入子树搬迁或默认向外层传播。
7. `childLaneSet` 在交互语义上按 Lane 处理；拖拽时的尺寸限制与 Lane 拖拽大小时保持同一套限制模型。

### 实施策略

- 首版实现默认采用当前文档里的局部收敛简化方案。
- 默认不兼容 `bpmn-js` 那种“从 lanes root 递归收集全部 Lane，再按几何邻接阈值决定调整对象”的 resize 模式。
- 如果后续在真实交互、回归测试或嵌套场景中出现当前方案无法稳定覆盖的异常案例，再按具体问题选择性吸收 `bpmn-js` 的处理逻辑，而不是预先整体照搬。

按这个边界，当前文档里的主模型、删除分配规则、贴边判定规则、`childLaneSet` 最小尺寸语义和传播边界已经基本收敛，可以进入实现设计阶段。
